## Problem Statement

Build a small open-source React component that recreates one specific effect well: a layered, dithered, canvas-based hero where foreground and background image or GIF layers can be independently interactive or passive.

This is not a startup wedge or a general creative-coding platform. It is a fun, focused package for developers who want a Browserbase-like interactive hero without rebuilding the canvas, dithering, filter, and React lifecycle plumbing themselves.

## What Makes This Cool

The "whoa" is the layered illusion: one visual layer can sit calmly in front while another reacts behind it. The effect feels playful because it is not just a shader slapped onto an image. It has depth, texture, dithering, and pointer response.

The component should feel good in the first 60 seconds:

```tsx
<DitheredParticleCanvas
  foreground="/foreground.png"
  background="/background.gif"
  interactiveLayer="background"
/>
```

Advanced customization matters, but the first example must stay tiny.

## Constraints

- Keep V1 centered on one use case: a two-layer dithered hero.
- Optimize for React developers, not creative-coding experts.
- Performance is part of the product. The component should not re-render React on every frame.
- The canvas renderer should work without React, but only as an internal architecture boundary.
- Dithering, filters, and interactivity should be configurable without turning the public API into a rendering engine DSL.
- GIF support is desirable, but it must not sink the first usable release.
- Browser compatibility matters. Off-main-thread rendering is an optimization, not a requirement.

## Premises

1. The package wins by making one polished effect easy, not by supporting every possible canvas effect.
2. React should own lifecycle and props; the renderer should own animation, pointer state, pixels, and cleanup.
3. The default path should be simple enough for a README hero example, with escape hatches for serious customization.
4. Ordered dithering should be the default because it is visually close to the reference and more stable under animation than error diffusion.
5. WebGL2 should be the performance target, but Canvas2D should ship first to prove the API and visual model.
6. GIF support should be adapter-based so implementation can start with still images and add animated sources without changing the public layer API.

## Approaches Considered

### Approach A: Minimal Two-Layer Canvas2D MVP

Build the React API, renderer lifecycle, source loading, Canvas2D rendering, ordered dithering, basic filters, and pointer interaction first.

Effort: M
Risk: Low

Pros:

- Fastest path to a usable open-source package.
- Proves the public API before shader/backend complexity hardens.
- Easier to test because CPU output can be snapshot-tested directly.

Cons:

- May miss the final performance target for large GIFs or high-DPI canvases.
- Some shader-friendly design decisions may need adjustment later.
- Browserbase-level smoothness may not be reachable until WebGL2 lands.

Reuses:

- Browser platform APIs: `<canvas>`, `ImageData`, `ResizeObserver`, `requestAnimationFrame`, pointer events.

### Approach B: Performance-First WebGL2 Core

Start with WebGL2 textures and shaders for dither, filters, palette mapping, and displacement. Build Canvas2D as a fallback after the GPU path is stable.

Effort: L
Risk: Medium

Pros:

- Best chance of hitting 60 FPS for large hero canvases early.
- Dither and filters map naturally to fragment shaders.
- Forces performance discipline from the first implementation.

Cons:

- Slower path to the first open-source release.
- Harder tests because GPU output needs tolerance-based image comparison.
- More intimidating for contributors.

Reuses:

- WebGL2, shader pipeline, texture upload, feature detection.

### Approach C: Preset-First React Component

Ship a highly opinionated `<DitheredHero />` with a few presets, then expose the lower-level renderer later.

Effort: S
Risk: Medium

Pros:

- Tiny API and fast README demo.
- Strong first impression because defaults can be tuned tightly.
- Avoids premature plugin architecture.

Cons:

- Too narrow if developers immediately want custom filters, custom palettes, or per-layer controls.
- Could paint the API into a corner if presets hide the real layer model.
- Does less to prepare the engineering plan for extensibility.

Reuses:

- Same renderer concepts as Approach A, but with a smaller public surface.

## Recommended Approach

Choose **Approach A: Minimal Two-Layer Canvas2D MVP**, with one explicit architectural constraint: design the renderer interface so WebGL2 can replace the backend without changing the React API.

This matches the stated goal: "pretty simple" and "one use case." It gives the engineering review a concrete scope boundary:

```text
React API
  -> renderer lifecycle
  -> two layer configs
  -> Canvas2D ordered dither
  -> basic filters
  -> pointer field
  -> tests + playground
```

Then WebGL2, workers, and GIF decoding become performance upgrades, not prerequisites for proving the package.

## Open Questions

- Should V1 support GIFs in the first release, or should V1 accept the same `src` shape but initially process only static images and first GIF frames?
- Should the public API prefer simple props (`foreground`, `background`, `interactiveLayer`) with advanced `layers` as an escape hatch?
- Should filter plugins exist in V1, or should V1 only expose built-in filters and reserve plugins for V2?

## Success Criteria

- A developer can create the reference-style two-layer hero in under 10 lines.
- Pointer movement affects only the configured interactive layer.
- React does not re-render on animation frames.
- The default visual preset looks intentionally dithered, not accidentally pixelated.
- Canvas2D MVP has deterministic snapshot tests for dithering and filters.
- The architecture has a documented backend boundary for later WebGL2.
- The playground includes the exact target demo: passive foreground over interactive background.

## Distribution Plan

The package should ship through npm.

Initial package target:

```bash
npm install @dithered-particle-canvas/react
```

Distribution requirements:

- Build ESM output for `@dithered-particle-canvas/core` and `@dithered-particle-canvas/react`.
- Mark React as a peer dependency.
- Include package exports for SSR-safe imports.
- Add a GitHub Actions release workflow before the first public npm release.
- Publish docs and playground examples from the repo, but keep npm package usage independent of the docs site.
